# /pdf-bookmarks

OCR한 책 PDF는 텍스트는 있지만 목차 기반 책갈피가 없다. 이 스킬은 책의 **인쇄된 목차 페이지**를 읽어 PDF outline(책갈피)을 생성한다. 사용자가 목차 따라 하나씩 손으로 책갈피 다는 노가다를 없앤다.

도구: **이 스킬 폴더의 `scripts/pdf_toc.py`** (PyMuPDF 기반). 아래 `$SCRIPT`는 이 SKILL.md와 같은 폴더의 `scripts/pdf_toc.py`를 가리킨다(위치 하드코딩 금지 — 스킬이 어디에 설치돼 있든 상대참조).

### 사전 요구 (먼저 확인)
- **필수**: Python 3.9+ 와 `pymupdf`(`pip install pymupdf`). 책갈피 전 기능이 이걸로 동작(전 OS 동일).
- **OCR가 필요할 때만**(생스캔): 플랫폼별 백엔드 — `auto`/`force`(tesseract, `ocrmypdf` 설치 시) 전 OS · `vision`(macOS 내장) · `winocr`(Windows 내장, 한국어 OCR 언어팩). 아래 OCR 스위치 참조.
- **이미 OCR된 PDF**(Acrobat·ABBYY·PDF Expert 등 무엇으로든)는 OCR 불필요 → `--ocr off`(기본).
- 샌드박스/제한 환경(tesseract·Vision·Windows OCR 없음)에서는 `--ocr off`만 쓰고 입출력 경로는 그 환경 규칙을 따른다.

## 핵심 개념

- 목차엔 `제목 …… 인쇄쪽번호`가 있다. 이걸 파싱해 outline으로 쓴다.
- **함정: 인쇄 쪽번호 ≠ PDF 물리 페이지** (표지·서문 때문에 상수 오프셋 존재). `--offset auto`가 첫 제목을 본문에서 검색해 오프셋을 1회 산출한다.
- 파싱·오프셋의 품질은 **목차 페이지 OCR 품질**이 좌우한다. 점선 리더·쪽번호가 뭉개지면 파싱이 흔들린다.

## 워크플로 (이 순서로 진행)

`$SCRIPT` = 이 스킬 폴더의 `scripts/pdf_toc.py` (실제 절대경로로 치환해 실행)

1. **목차 페이지 찾기** — 앞부분 텍스트를 덤프해 "목차/Contents/차례"가 몇 쪽(PDF 물리 페이지)에 있는지 눈으로 확인한다.
   ```
   python3 $SCRIPT extract <파일.pdf>
   ```
   - 텍스트가 거의 안 나오면 **생스캔**(텍스트 레이어 없음)이다 → `--ocr auto` 붙여 다시: `extract <파일.pdf> --ocr auto`.

2. **미리보기(dry-run)** — 찾은 목차 페이지 범위로 파싱 결과를 사용자에게 보여주고 확인받는다. 조용히 적용하지 말 것.
   ```
   python3 $SCRIPT apply <파일.pdf> --toc-pages <범위> --dry-run
   ```
   - 항목이 0개거나 레벨/쪽이 이상하면: 범위를 조정하거나, 오프셋 자동감지 실패 시 `--offset <정수>` 수동 지정.

3. **적용** — 확인되면 dry-run을 빼고 실행.
   ```
   python3 $SCRIPT apply <파일.pdf> --toc-pages <범위>
   ```
   - 결과는 **항상 별도 파일 `<원본>-marked.pdf`**로 저장(원본 불변, 클라우드 되돌림 문제 회피). 다른 경로/이름은 `-o <출력.pdf>`.
   - **목차/차례/Contents 페이지**는 자동으로 최상위(레벨1) 책갈피로 추가된다. 폰트 모드에서 첫 부(part) 이전의 앞머리 섹션(머리말 등)도 최상위로 승격.
   - **스캔본은 `--page-labels`를 함께 붙여라**(권장): 뷰어의 페이지 카운터를 책 인쇄쪽번호에 맞춘다. 아래 참고.

## 저품질 OCR 폴백: `--from-list` (사람이 목차를 읽어 제공)

**증상**: 생스캔이 흐려 OCR 품질이 낮음 → 목차 쪽번호가 뭉개져 정규식 불가 + OCR본은 폰트가 노이즈라 `--by-font`도 불가(스캔 OCR엔 신뢰할 폰트 메타데이터 없음). 두 자동 모드가 다 실패한다.

**해법**: 목차 페이지를 **이미지로 렌더해 눈으로 읽어**(OCR 텍스트 말고 렌더 이미지 — 추측 금지, 실제로 보고 옮긴다) 목록 파일을 만든 뒤 `--from-list`로 적용. 오프셋은 자동감지되어 인쇄쪽→PDF쪽 변환.

1. **목차 페이지 렌더** → 시각적으로 읽는다(150dpi 이상. 저해상도면 백지처럼 보일 수 있으니 올려라).
   ```
   python3 -c "import fitz; fitz.open('<파일>')[<목차0based>].get_pixmap(dpi=160).save('/tmp/toc.png')"
   ```
2. **목록 파일 작성** — 줄마다 `level | 제목 | 인쇄쪽`. 앞표지·부 디바이더 등 인쇄번호 없는 항목은 `=N`(절대 PDF쪽). 예:
   ```
   1 | 차례 | =9
   1 | 들어가는 말 | 8
   1 | 1부 | =23
   2 | 1 외계 지성의 탄생 | 22
   ...
   ```
3. **적용** (오프셋 자동감지 + 라벨):
   ```
   python3 $SCRIPT apply <파일.pdf> --from-list toc.txt --page-labels
   ```
   - 착지 페이지를 2~3개 **이미지로 검증**(장 제목 페이지가 맞는지). 오프셋 자동감지가 틀리면 `--offset N`(=PDF−인쇄) 수동.
- 생스캔이면 앞에 `--ocr auto`로 텍스트 레이어 먼저 입히거나, OCR본(`_ocr.pdf`)을 대상으로 실행한다.

## 페이지 라벨 (`--page-labels`) — 뷰어 번호 = 책 인쇄번호

스캔본은 앞표지·목차가 물리 페이지를 차지해 **PDF 물리쪽 ≠ 책 인쇄쪽번호**(예: PDF 30쪽인데 책엔 28쪽). 책갈피 착지는 정확하지만 뷰어 카운터와 인쇄번호가 어긋나 보인다.

`apply`에 `--page-labels`를 붙이면 PDF page label을 심어 뷰어가 인쇄번호를 표시한다. **오프셋(PDF−인쇄)은 책마다 다르므로 자동감지**한다 — 절대 하드코딩 금지(실측: 책마다 1·2·17 등 제각각). 원리: 회전·포맷 무관하게, PDF쪽이 1 늘 때 같이 1 느는 수만 진짜 쪽번호라 그 오프셋이 압도적으로 누적됨(연도·'총서 17' 같은 상수는 탈락). 앞머리 오프셋쪽은 로마숫자(i…), 본문은 아라비아(1…)로 라벨.

```
python3 $SCRIPT apply <파일.pdf> --by-font --font-levels "..." --body-start N --page-labels
```
- 자동감지 신뢰도가 낮으면 라벨을 생략하고 알린다 → `--label-offset N`으로 수동 지정 가능(N = PDF물리쪽 − 책인쇄쪽, 본문 한 페이지에서 계산).
- 저장 후 `page.get_label()`로 한두 개 확인(예: 1장 착지 PDF쪽 라벨 == 책 목차의 그 장 쪽번호).

## 정규식이 안 될 때: 폰트 기반 모드 (`--by-font`)

**증상**: `apply --dry-run` 결과가 0개이거나 러닝푸터("N | 책제목")만 잡힘. `extract`로 목차를 보면 **제목이 한 뭉치, 쪽번호가 따로 한 뭉치**로 분리돼 있음(2단 목차). 밀집 학술서에 흔함. 단일줄 정규식으로는 구조적으로 불가.

**해법**: 깨진 쪽번호를 버리고 **본문에서 큰 폰트 헤딩을 스캔**한다.

1. **폰트 크기 확인** — 본문 표본에서 크기 히스토그램을 본다.
   ```
   python3 $SCRIPT extract <파일.pdf> --pages <본문범위> --fonts
   ```
   부/장/본문이 서로 다른 크기로 뜬다(예: 부 40, 장 18, 본문 10).
2. **밴드 지정해 추출** — 크기 구간→레벨 매핑. `--body-start`는 목차 뒤 본문 시작쪽.
   ```
   python3 $SCRIPT apply <파일.pdf> --by-font --font-levels "36-45:1,16-23:2" --body-start 17 --dry-run
   ```
3. **착지 페이지 검증 후 저장** — dry-run 목록의 페이지가 실제 장 시작인지 몇 개 확인(할루시네이션 방지), 맞으면 `--dry-run` 빼고 실행.

주의:
- 필터가 순수 쪽번호 조각·장식페이지 OCR쓰레기(`|`≥2)·부 부제 중복·레벨2의 너무 짧은 조각을 자동 제거한다.
- 2단 목차는 OCR이 좌우 컬럼을 섞어 뽑아서 **제목 단어 순서가 뒤섞이거나 장번호가 누락**될 수 있다. **페이지는 정확**하니 내비게이션엔 문제없음. 제목 정밀 교정이 필요하면 사용자에게 알리고 별도 처리.

## OCR 스위치 (`--ocr`, extract·apply 공통)

두 플로우를 한 옵션으로 선택한다:

| 값 | 동작 | 플랫폼 | 언제 |
|---|---|---|---|
| `off` (기본) | OCR 안 함 | 전부 | 이미 OCR된 파일(어떤 툴로 했든). |
| `auto` | 텍스트 없을 때만 tesseract | 전부 | 생스캔. 이미 텍스트 있으면 건너뜀. |
| `force` | 무조건 tesseract 재OCR | 전부 | 기존 OCR이 엉망일 때. |
| **`vision`** | **macOS Vision**(고품질) → 검색가능 PDF 재구성 | **macOS** | 로컬 Mac 생스캔. 한글 품질 최상. |
| **`winocr`** | **Windows.Media.Ocr**(고품질) → 검색가능 PDF 재구성 | **Windows** | Windows 생스캔. Vision 대칭. |

**플랫폼별 OCR 전략**: 책갈피 기능(정규식·`--by-font`·`--from-list`·페이지라벨·오프셋감지)은 **전 OS 동일**(순수 PyMuPDF). OCR만 플랫폼별 최적 백엔드를 고른다 — macOS=`vision`, Windows=`winocr`, Linux/그 외=`auto`(tesseract). 플랫폼이 안 맞으면 각 백엔드가 **명확한 에러**로 다른 옵션을 안내한다(크래시 아님).

**`vision`/`winocr` 공통 구조**: 각 페이지 렌더 → 네이티브 OCR(줄단위 텍스트+박스) → 시스템 한글폰트 임베드 투명 텍스트로 검색가능 PDF 재구성(회전 자동 반영, JPEG 압축). ~0.9초/쪽. tesseract가 뭉개는 흐린 스캔도 이 둘은 깨끗 → OCR 후 자동 모드가 살아난다.
- `vision`: `scripts/visionbox.swift`(swiftc 머신별 1회 컴파일), 폰트 AppleSDGothicNeo. macOS 내장, 무설치.
- `winocr`: `scripts/winocr.ps1`(PowerShell+WinRT), 폰트 맑은 고딕(malgun.ttf). **요구: Windows 10/11 + 한국어 OCR 언어팩**(설정>언어>한국어>선택적 기능>광학 문자 인식). ⚠️ **이 백엔드는 Windows 실기 검증 필요**(개발은 macOS에서 이뤄져 미검증).

- OCR가 실행되면 결과를 `<원본>_ocr.pdf`로 저장하고 책갈피는 그 파일에 기록한다(원본 불변).
- 언어 기본값 `--lang kor+eng`. 다른 언어면 변경.
- **권장**: 평소 PDF Expert로 OCR하던 책은 `--ocr off`(기본) 그대로 — 목차 페이지에서 tesseract보다 품질이 낫다. `--ocr auto`는 "OCR 안 한 생스캔 예외" 대비용.

## 주의

- 여러 파일을 한 번에 요청받으면 파일별로 1~3단계를 반복한다(목차 위치·오프셋이 책마다 다름).
- 오프셋이 중간에 바뀌는 책(부/편마다 페이지 리셋)은 상수 오프셋으로 안 맞는다 → 그 사실을 알리고 부분 적용하거나 수동 오프셋으로 나눠 처리.
- 파괴적 아님: 항상 `-marked.pdf` 새 파일 생성, 원본 불변.
- **클라우드 동기화 폴더 교훈**: 기존 대용량 PDF를 제자리 증분 저장하면 동기화 클라이언트(Google Drive/Dropbox/iCloud/OneDrive)가 클라우드 원본으로 되돌려 책갈피가 사라진다(실제 발생). 그래서 기본을 새 파일 저장으로 바꿨다. 저장 후 `get_toc`로 재확인하고, 중요하면 비-동기 위치(~/Desktop) 사본도 남긴다.
- **페이지 번호 주의**: 책갈피는 PDF 물리 페이지를 가리킨다. 스캔본은 앞표지·목차가 물리 페이지를 차지해 **PDF 페이지 ≠ 책 인쇄 쪽번호**(예: 이 책은 PDF = 인쇄+2). 책갈피 착지는 정확하지만 뷰어의 페이지 카운터와 인쇄 번호가 다를 수 있다. 뷰어가 인쇄 번호를 표시하게 하려면 PDF page label을 별도로 설정해야 한다(현재 미구현).
